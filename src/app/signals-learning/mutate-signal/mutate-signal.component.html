mutate() — Removed in Angular 17+

📖 Why it was removed & how to migrate to update()

Breaking change (Angular 17): signal.mutate() was removed. It allowed in-place mutation of signal values (arrays, objects) without creating a new reference. This broke Angular's change detection which relies on reference equality.
❌ Old way — mutate() (Angular 16) Mutates array in-place — change detection may miss this 
items = signal(['Apple', 'Banana']);

// mutate() — modifies the SAME array reference
// Angular may NOT detect this change reliably
this.items.mutate(arr => {
  arr.push('Mango');      // ← in-place push
});
cancel Removed — unreliable change detection, confusing semantics
✅ New way — update() (Angular 17+) Returns a NEW array reference — change detection works perfectly 
items = signal(['Apple', 'Banana']);

// update() — returns a NEW array (spread operator)
// Angular ALWAYS detects the new reference
this.items.update(arr => [
  ...arr,         // copy existing items
  'Mango'         // ← add new item
]);
check_circle Correct — immutable update pattern, predictable change detection
Common migration patterns
Operation ❌ Old mutate() pattern ✅ New update() pattern
Add to array signal.mutate(a => a.push(x)) signal.update(a => [...a, x])
Remove from array signal.mutate(a => a.splice(i, 1)) signal.update(a => a.filter(i => i !== x))
Update object property signal.mutate(o => o.name = 'New') signal.update(o => ({...o, name: 'New'}))
Update item in array signal.mutate(a => a[0].qty++) signal.update(a => a.map(i => i.id === id ? {...i, qty: i.qty+1} : i))
lightbulb
Golden Rule: Signals use reference equality to detect changes. Always return a new object or array reference from update() — never modify the existing value in-place.
