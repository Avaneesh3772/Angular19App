catchError() · throwError() · retry()

🛡️ E-Commerce Checkout Resilience — recover gracefully from failures

Problem: A checkout system must survive payment failures, dead APIs, and flaky networks — without crashing the app or showing a blank screen.

What we built: A resilience hub with three real failure scenarios. Each accordion shows how RxJS catches errors and keeps the stream alive.

throwError() — creates an Observable that immediately errors (simulates a failed payment gateway).

catchError() — intercepts the error and returns a new Observable (fallback data or backup API). The stream continues.

retry(n) — re-subscribes up to n times on failure — useful for transient network glitches.

Golden rule: catchError must always return an Observable — use of(fallback), a backup API, or EMPTY.

} When to use each error-handling operator
OperatorWhat it doesIn this checkout hub
throwError() Creates an Observable that errors immediately Simulate payment gateway timeout (HTTP 504)
catchError() Catches error, returns replacement Observable Load ordersData.json or employeeData.json instead of crashing
retry(n) Re-subscribes up to n times on failure Retry flaky inventory API before giving up

Common catchError patterns

catchError(err => of([]))                 // empty fallback
catchError(err => this.backupApi())       // switch to backup API
catchError(err => throwError(() => err))  // rethrow to parent
